Modelo de Datos

Este es el criterio más importante. El modelo de datos de la base de datos debe alinearse con la estructura de los datos de la aplicación. Si sus datos son altamente interconectados, una base de datos de grafos como Neo4j es la opción natural. Si trabaja con objetos complejos y flexibles, una base de datos documental como MongoDB será más intuitiva. Si su principal desafío es la ingesta masiva de datos de series temporales, una columnar como Cassandra es una fuerte candidata.
🔍

Patrones de Consulta y Carga de Trabajo

¿Su aplicación realizará muchas más lecturas que escrituras, o viceversa? ¿Las consultas serán simples o complejas? Las bases de datos clave-valor son imbatibles para lecturas y escrituras simples por clave. Las bases de datos documentales ofrecen una gran flexibilidad para consultas ad-hoc. Las bases de datos columnares están optimizadas para cargas de trabajo con muchas escrituras y lecturas analíticas sobre columnas específicas.
🖥️

Requisitos de Escalabilidad y Rendimiento

¿Cuántos usuarios concurrentes espera? ¿Cuál es el volumen de datos previsto? ¿Qué latencia es aceptable? Todas las bases de datos NoSQL están diseñadas para escalar horizontalmente, pero algunas, como Cassandra, están construidos para una escala masiva y una disponibilidad continua desde el principio.
⚖️

Consistencia vs. Disponibilidad (Teorema CAP)

Como se discutió en la Unidad 1, debe decidir qué es más crítico para su aplicación. Para un sistema de comercio electrónico, la disponibilidad puede ser más importante que la consistencia estricta. En este caso, un sistema AP como Cassandra es una buena opción. Para un sistema bancario, la consistencia es primordial, lo que podría favorecer a un sistema CP o a un SGBD tradicional.
